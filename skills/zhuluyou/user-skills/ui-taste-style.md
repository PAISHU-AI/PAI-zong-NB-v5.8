## Purpose

Capture the user's UI and product design expectations for desktop and Web software.

## Trigger Conditions

- Use before UI design, redesign, beautification, visual QA, or implementation of product screens.
- Use when working on desktop/Web tools, dashboards, API-driven apps, admin panels, or SaaS-like screens.

## Behavior Rules

- Avoid generic AI-looking UI.
- Prefer professional product polish over decorative gimmicks.
- Use mature external design/taste skills when useful.
- Consider desktop/Web tool usability, information density, state handling, and maintainability.
- Use screenshots or browser validation when important UI changes are implemented.
- In V5.0, pair UI work with `skill-router-style.md` so 编码师 selects the right UI professional skills.
- Stable UI taste changes should be routed through `user-profile-growth-style.md`.

## Forbidden Behaviors

- Do not rely on template-like default SaaS layouts.
- Do not add random gradients, shadows, or motion without improving hierarchy or usability.
- Do not ignore loading, empty, error, disabled, success, and permission states.

## Output / Acceptance Checks

- UI has clear hierarchy, spacing, typography, and interaction states.
- Design direction is intentional rather than generic.
- Implementation follows existing project patterns when present.
